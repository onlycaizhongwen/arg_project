# 数据清洗与 RAG 服务架构实施计划

## 目标

基于需求与设计文档，将“数据清洗与 RAG 服务架构”拆成可执行阶段，先交付可闭环的 MVP，再逐步增强接入源、检索质量、治理能力和运维能力。

## 前置条件

- 确认首期技术栈：统一 Python，FastAPI 控制面 + Python Worker。
- 确认基础设施可用性：关系库、对象存储、Redis、消息队列、向量库、模型服务。
- 确认首批数据源和文件格式优先级。
- 确认模型调用方式：本地私有化、企业模型平台或外部 API。
- 确认权限模型：无权限、租户级、文档级、段落级。

## 阶段 0：选型验证

1. 验证对象存储上传、读取和元数据保存。
2. 验证 2 到 3 类核心文件格式解析，例如 PDF、Word、CSV。
3. 验证 Embedding 服务调用和批量吞吐。
4. 验证向量库写入、过滤查询和 TopK 检索。
5. 验证关键词检索是否需要独立 ES/OpenSearch。
6. 验证检索漏斗：召回候选数、粗排耗时、去重/打散效果和精排候选规模。

验收标准：

- 能将一份样例文档解析成文本块并写入向量库。
- 能按查询召回相关片段，并通过粗排、去重/打散、精排后返回来源信息。
- 能给出向量库和关键词检索的最终首期选型建议。

## 阶段 1：MVP 主链路

1. 建立元数据模型：数据源、文档、文档版本、清洗任务、文本块、向量记录。
2. 实现文件接入 API：上传、创建任务、查询任务状态。
3. 实现异步任务队列：接入任务投递、Worker 消费、失败重试。
4. 实现基础解析与清洗：格式解析、编码标准化、简单噪声过滤、空内容校验。
5. 实现文本切分：按章节/段落和长度切分，生成 chunk 元数据。
6. 实现 Embedding 与向量入库：批量调用模型、写入向量库、记录索引状态。
7. 实现语义检索 API：query embedding、向量 TopK、权限/知识库过滤、结果返回。
8. 实现基础粗排和候选截断：先用向量内积或融合分数将候选压缩到可控规模。
9. 实现任务监控最小能力：状态、错误原因、阶段耗时、重试入口。

验收标准：

- 文件上传后可自动完成清洗、切分、向量化和索引。
- 检索接口可返回片段、文档、页码/章节、分数和元数据。
- 检索接口不会将大规模候选直接提交给精排模型，候选规模可配置、可观测。
- 任一阶段失败可定位到任务、文档版本和错误阶段。

## 阶段 2：混合检索与质量增强

1. 增加关键词索引：ES/OpenSearch 或可接受的全文检索方案。
2. 实现混合召回：语义召回 + 关键词召回 + 多路召回来源记录。
3. 实现粗排：双塔/DSSM/FM/DNN 或向量内积，将大候选集压缩到千级。
4. 实现业务干预：SimHash/MinHash 去重，MMR/DPP 打散，权限、时效、质量过滤。
5. 接入重排模型：仅对约 50 到 100 个候选精排，支持超时降级、开关配置。
6. 增强清洗规则：页眉页脚、重复段落、特殊字符、表格处理。
7. 增强质量校验：重复率、结构完整性、编码异常、规则一致性。
8. 增加索引重建：按文档、知识库、索引版本重建。

验收标准：

- 混合检索结果优于单纯语义检索的样例集评测。
- 10 万级候选经过粗排和业务干预后，只将约 50 到 100 个候选提交给精排。
- 重排服务不可用时检索接口可降级返回。
- 索引重建不会暴露半完成版本。

## 阶段 3：多源接入与资产治理

1. 增加数据库同步接入：MySQL/Oracle 增量或批量同步。
2. 增加流式接入：日志、消息队列事件。
3. 增加第三方系统接入：OA/CRM/企业知识库。
4. 完善数据资产管理：原始文档库、清洗文本库、向量索引库、元数据检索。
5. 完善血缘追踪：数据源到文档版本、chunk、vector、索引版本的链路。
6. 增加策略管理：清洗策略、切分策略、模型配置、索引策略。

验收标准：

- 至少一种非文件接入源跑通。
- 可以从检索结果反查到原始文档、清洗任务和索引版本。
- 策略调整后可以对指定范围重跑清洗或重建索引。

## 阶段 4：业务服务扩展

1. 智能客服接入：检索接口与问答编排系统对接。
2. 企业知识搜索：支持多知识库、多条件过滤、排序和高亮。
3. 合规审查助手：增加规则标签、审查类型和证据片段返回。
4. 文档摘要服务：基于清洗文本和分块结果生成摘要。
5. 内容推荐：基于向量相似度和业务标签推荐相关文档。

验收标准：

- 至少一个业务应用完成端到端接入。
- RAG 检索服务的输出满足业务端展示和追溯要求。

## 验证方式

- 单元验证：解析器、清洗规则、切分策略、模型接口、向量写入。
- 集成验证：文件上传到可检索的全链路。
- 数据质量验证：样例文档解析完整性、重复过滤准确性、chunk 可读性。
- 检索质量验证：构造查询集，评估召回率、粗排淘汰效果、去重/打散效果、TopK 命中、精排效果。
- 稳定性验证：重复上传、任务重试、模型超时、向量库写入失败。
- 权限验证：不同租户或权限上下文下不可越权召回。

## 风险与回滚

- 解析质量不稳定：保留原始文件和中间产物，允许替换解析器后重跑。
- 模型服务不可用：Embedding 任务进入可重试失败状态；检索粗排和精排可降级。
- 候选集过大拖垮精排：通过 `recall_size`、`pre_rank_size`、`rerank_size` 强制限制候选规模。
- 向量库索引异常：使用索引版本管理，保留旧版本可见，新版本构建完成后切换。
- 数据重复：通过 checksum、版本号和阶段幂等键控制。
- 权限遗漏：检索前置权限上下文校验，向量过滤和结果返回双重校验。

## 检查点

- CP1：完成选型验证后，确认向量库、关键词检索和模型部署方式。
- CP2：MVP 主链路完成后，确认是否进入混合检索增强。
- CP3：多源接入前，确认首批外部系统和数据同步边界。
- CP4：业务应用接入前，确认 API 契约、权限模型和追溯字段。

## 当前建议的最小落地顺序

1. 文件接入 + 元数据管理。
2. 异步清洗任务 + 基础解析。
3. 文本切分 + Embedding + 向量库。
4. 语义检索 API。
5. 混合检索 + 粗排 + 业务干预 + 精排。
6. 数据资产、血缘和索引治理。
