# 数据清洗与 RAG 服务下一步执行计划

## 当前结论

已完成需求分析、技术选型与总体设计。下一步应进入“选型验证 + MVP 骨架准备”，目标是在最短路径内跑通一条可演示闭环：

```text
上传文档 -> 保存元数据 -> 异步清洗 -> 文本切分 -> Embedding -> 向量入库 -> 语义召回 -> 基础粗排/候选截断 -> 返回来源片段
```

## 需要先确认的 5 个决策

| 决策项 | 推荐选择 | 备选 | 影响 |
| --- | --- | --- | --- |
| 后端技术栈 | FastAPI 控制面 + Python Worker | 单体 FastAPI | 决定工程结构、任务调用方式和模型集成成本 |
| 向量库 | Qdrant 或 Milvus | pgvector | 决定部署复杂度、过滤能力和后续扩展 |
| 消息队列 | RabbitMQ | Kafka | 决定异步任务模型和吞吐能力 |
| 文件存储 | MinIO | 本地文件目录 / OSS | 决定原始文件与中间产物管理方式 |
| 模型服务 | 本地/企业 Embedding 服务抽象接口 | 外部 API | 决定调用封装、超时、成本和私有化能力 |

## 推荐第一阶段范围

### 必做

1. 项目工程初始化。
2. 元数据表设计：数据源、文档、文档版本、清洗任务、文本块、向量记录。
3. 文件上传接口。
4. 任务状态查询接口。
5. 清洗 Worker 骨架。
6. PDF / Word / CSV 中至少 2 类格式解析验证。
7. 文本切分策略。
8. Embedding 调用适配层。
9. 向量库写入和 TopK 查询。
10. 语义检索 API。
11. 基础粗排和候选规模限制；MVP 不接 Cross-Encoder 精排。

### 暂不做

- 多源数据库同步。
- OA/CRM 第三方系统集成。
- 复杂 OCR 和音视频转写。
- 多轮问答编排。
- 完整后台管理页面。
- 领域模型微调。

## 建议任务拆分

### Step 1：工程与基础设施确认

- 确认使用 Java、Python 或混合架构。
- 确认本地开发环境依赖：数据库、Redis、MQ、向量库、对象存储。
- 输出 `docker-compose` 或本地依赖说明。

完成标准：

- 开发环境可以启动。
- API 服务和 Worker 可以连通基础设施。

### Step 2：元数据模型与接口契约

- 设计并落地 6 类核心对象。
- 定义文件上传、任务查询、检索接口。
- 补充状态机和错误码。

完成标准：

- 可以创建文档版本和清洗任务。
- 可以查询任务状态。

### Step 3：文件接入与清洗 Worker

- 上传文件保存到对象存储或本地存储。
- 投递清洗任务。
- Worker 消费任务并执行解析。
- 输出清洗文本。

完成标准：

- 上传样例文档后，能得到清洗后的纯文本或结构化文本。

### Step 4：切分、Embedding 与向量入库

- 将清洗文本切分为 chunk。
- 关联文档、版本、页码、章节等元数据。
- 调用 Embedding。
- 写入向量库并记录 `vector_record`。

完成标准：

- 文档处理完成后，向量库中有可查询的 chunk 向量。

### Step 5：语义检索闭环

- 提供检索 API。
- 查询文本生成向量。
- 从向量库召回 TopK。
- 使用基础粗排或融合分数控制候选规模。
- MVP 不接 Cross-Encoder 精排，避免引入过早的模型延迟和成本。
- 返回内容片段、来源文档、分数和元数据。

完成标准：

- 使用样例问题可以检索到目标文档片段。
- 返回结果可追溯到文档和文本块。
- 候选规模需要可配置；MVP 不接精排模型，也不为后续精排留下大候选直传路径。

## 近期验收 Demo

准备 3 份样例文档：

1. 企业制度文档。
2. 产品说明文档。
3. FAQ 或客服知识文档。

演示路径：

```text
上传样例文档
-> 查看任务状态从 UPLOADED 到 INDEXED
-> 输入问题
-> 返回相关片段
-> 展示片段来源、分数、文档 ID、chunk ID
```

## 主要风险

- 如果向量库和模型服务还未确定，建议先用可替换适配层，不把实现绑死。
- 如果没有对象存储，MVP 可先用本地目录，但接口层保留 `object_uri`。
- 统一 Python 后，仍建议拆分 FastAPI 控制面和 Python Worker，避免 API 请求线程直接承担文档解析和模型调用重任务。
- 如果短期没有 ES/OpenSearch，先做语义检索，混合检索放到第二阶段。
- 如果候选集达到万级或十万级，必须先做粗排和业务干预，再进入精排。

## 下一步建议

先确认以下选择：

1. 首期采用“FastAPI 控制面 + Python Worker”。
2. 向量库选 Qdrant、Milvus 还是 pgvector。
3. 本地是否允许用 Docker Compose 拉起 PostgreSQL、Redis、RabbitMQ、MinIO、向量库。
4. Embedding 模型使用本地模型、企业模型平台还是外部 API。
5. MVP 是否只做文件上传，不做数据库同步和第三方系统集成。
