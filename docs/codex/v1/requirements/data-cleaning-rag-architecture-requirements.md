# 数据清洗与 RAG 服务架构需求分析

## 背景

当前需求来自“数据清洗与 RAG 服务架构”图，目标是建设一套面向企业知识场景的数据接入、清洗、资产管理、向量化、检索增强与对外服务能力。系统需要支撑智能客服、企业知识搜索、合规审查助手、内容推荐、文档摘要等业务应用。

仓库当前尚无业务代码，本需求文档先作为 v1 规划基线，用于后续技术设计、实施计划和实现拆分。

## 目标

1. 建立统一的数据接入能力，支持文件、实时流、数据库同步和第三方系统集成。
2. 建立可追踪的数据清洗流水线，完成格式解析、噪声过滤、质量校验、结构化标准化与失败重试。
3. 建立数据资产管理能力，沉淀原始文档、清洗后文本、向量索引、元数据和血缘记录。
4. 建立语义切分与向量化能力，支持文档分块、元数据关联、Embedding 生成和领域模型扩展。
5. 建立向量存储与检索排序能力，支持语义检索、关键词检索、混合召回、粗排、业务干预、精排和索引优化。
6. 对外提供 RAG 检索服务 API，支撑 Web、APP、企业微信或内部系统接入。
7. 提供横向治理能力，包括数据接入管理、清洗流程监控、模型与索引管理。

## 范围

### v1 必须覆盖

- 文件上传接入：PDF、Word、Excel、PPT、HTML、JSON、CSV、图片和音视频的元信息接入；音视频内容转写可作为扩展。
- API 接入网关：统一接收接入请求、鉴权、限流、任务创建和状态查询。
- 清洗任务编排：解析、过滤、标准化、质量校验、切分、向量化、入库、索引构建。
- 数据资产管理：原始文档、清洗文本、元数据、向量索引、处理状态、错误原因、血缘关系。
- RAG 检索服务：语义检索、关键词检索、混合召回、粗排、去重、打散、精排、结果返回。
- 基础观测：任务状态、失败重试、耗时、吞吐、索引构建进度、模型调用错误。

### v1 可选增强

- 数据库同步：MySQL、Oracle。
- 实时流式接入：日志、消息队列事件。
- 第三方系统集成：OA、CRM、知识库。
- 领域模型微调与质量校验模型。
- 自动摘要服务与内容推荐策略。

## 关键场景

### 场景 1：企业文档导入知识库

业务系统上传文档，系统创建接入任务，保存原始文件，解析正文与结构化元素，清洗噪声，生成分块，写入清洗文本库和向量库。用户随后可通过知识搜索或智能客服检索相关内容。

### 场景 2：知识更新与索引重建

当文档版本更新或删除时，系统需要识别影响范围，更新元数据、文本分块、向量数据和索引状态，避免检索到过期内容。

### 场景 3：RAG 漏斗式混合检索

业务应用提交查询，系统同时执行关键词检索和语义检索，可能产生万级到十万级候选片段。系统不能将大候选集直接提交给 Cross-Encoder 等精排模型，而应先经过“召回 -> 粗排 -> 业务干预 -> 精排”的漏斗式排序链路，将候选集逐层压缩和清洗后，再返回可追溯的片段、文档来源和置信信息。

### 场景 4：清洗失败追踪

当文件解析失败、模型调用失败、向量库写入失败或质量校验不通过时，系统记录失败阶段、错误原因、输入输出摘要，并支持人工重试或跳过。

## 功能需求

### FR-01 数据接入

- 支持多源接入任务统一建模。
- 支持文件上传、API 调用和批量导入。
- 为数据库同步、流式接入和第三方系统接入预留扩展接口。
- 每次接入生成唯一数据源、文档、版本和任务标识。

### FR-02 智能数据清洗

- 支持文档格式解析，保留标题、段落、页码、章节、表格和图片引用等结构信息。
- 支持噪声过滤，包括页眉页脚、重复内容、特殊字符、冗余格式和编码标准化。
- 支持质量校验，包括空内容、重复内容、结构异常、编码异常、字段缺失和规则不一致。
- 支持失败阶段记录、重试和人工干预。

### FR-03 语义切分与向量化

- 支持按段落、主题、固定长度和重叠窗口切分。
- 支持将分块与文档 ID、页码、章节、版本、权限标签等元数据关联。
- 支持调用 Embedding 模型生成向量。
- 支持领域模型或专用 Embedding 服务的替换。

### FR-04 向量存储与索引

- 支持向量写入、删除、更新和按租户/知识库隔离。
- 支持 HNSW、IVF 等索引策略配置。
- 支持索引构建、定期优化、降维或归档策略。
- 支持向量库与关系库、缓存的混合存储。

### FR-05 RAG 检索服务

- 支持语义检索、关键词检索、混合检索。
- 支持漏斗式排序链路，避免将大规模召回候选直接提交给精排模型。
- 支持粗排 Pre-ranking，用轻量模型或向量内积快速压缩候选集，例如从 10 万候选缩小到约 1000。
- 支持业务干预 Business Rules，对粗排候选进行去重、打散和必要的业务规则过滤。
- 支持精排 Reranking，仅对约 50 到 100 个高质量、多样化候选调用 Cross-Encoder 等重排模型。
- 返回结果需要包含片段内容、来源文档、页码/章节、分数、召回方式和可追溯元数据。
- 提供对外 API，并支持 Web、APP、企业微信和内部系统调用。

#### FR-05.1 召回 Recall

- 支持多路召回，包括语义向量召回、关键词召回、元数据过滤召回和业务规则召回。
- 召回阶段以“尽量不漏”为目标，可以产生较大候选集。
- 召回结果必须保留召回来源、原始分数、文档 ID、chunk ID、知识库 ID 和权限标签。

#### FR-05.2 粗排 Pre-ranking

- 粗排目标是用最小算力快速淘汰明显不相关候选。
- 支持双塔模型、DSSM、FM、DNN 或简单向量内积等轻量方案。
- 支持离线预计算知识片段向量，线上只计算 Query 向量后进行快速打分。
- 粗排输出应支持配置候选数量，默认建议从万级或十万级候选压缩到约 1000。
- 粗排耗时应纳入检索链路指标，目标为毫秒到几十毫秒级。

#### FR-05.3 业务干预 Business Rules

- 支持去重 Deduplication，使用 SimHash、MinHash 或相似度阈值去除内容高度相似的候选片段。
- 支持按文档、章节、知识库、业务类型和来源系统控制重复结果数量。
- 支持打散 Diversity，使用 MMR 或 DPP 保证候选结果覆盖不同主题、文档来源和答案类型。
- 支持业务规则过滤，例如权限过滤、时效过滤、禁用文档过滤、低质量片段过滤。
- 业务干预输出应支持配置候选数量，默认建议保留约 50 到 100 个候选进入精排。

#### FR-05.4 精排 Reranking

- 精排目标是对小规模高质量候选进行深度相关性判断。
- 支持 Cross-Encoder、重排模型服务或企业模型平台。
- 支持超时降级；精排不可用时，应返回粗排或业务干预后的候选结果。
- 精排结果需记录精排分数、模型版本、耗时和是否触发降级。

### FR-06 数据资产与治理

- 支持原始文档库、清洗后文本库、向量索引库、元数据管理和数据血缘追踪。
- 支持按知识库、租户、部门、业务域或权限标签管理数据。
- 支持清洗流程监控、任务审计和质量统计。

## 非功能需求

- 可扩展性：接入源、解析器、清洗规则、模型服务和向量库应可插拔。
- 可观测性：关键阶段必须有状态、耗时、错误码、重试次数和核心指标。
- 一致性：原文、文本块、向量和索引之间需要可追踪，可重建。
- 安全性：至少预留鉴权、租户隔离、权限标签和敏感信息处理扩展点。
- 幂等性：重复上传、重复回调、任务重试不能产生不可控重复数据。
- 可运维性：支持失败重试、任务补偿、索引重建和模型服务降级。
- 性能约束：检索排序必须采用漏斗式候选压缩，禁止将万级或十万级候选直接输入 Cross-Encoder 等高成本精排模型。

## 非目标

- v1 不直接实现完整问答大模型编排和多轮对话记忆。
- v1 不直接承诺所有格式的高精度解析，复杂 OCR、音视频转写先作为增强能力。
- v1 不直接建设完整数据中台，只覆盖 RAG 所需的数据资产闭环。
- v1 不直接做模型训练平台，只预留领域模型微调和模型服务替换能力。

## 完成标准

- 已形成接入、清洗、切分、向量化、检索排序、资产治理的端到端架构。
- 每类核心数据对象有明确归属、状态和血缘关系。
- 每个核心模块有职责边界、输入输出和失败处理策略。
- 技术选型有推荐方案、备选方案和取舍说明。
- 实施计划可拆分到 MVP、增强、治理三个阶段。

## 依赖

- 对象存储或兼容 S3/OSS 的文件存储。
- 关系型数据库，用于元数据、任务状态和审计。
- 缓存服务，用于热点配置、短期状态和检索缓存。
- 消息队列，用于异步清洗、向量化、索引构建和重试。
- 向量数据库或向量索引引擎。
- Embedding 模型服务、粗排模型服务和重排模型服务。

## 待确认问题

- 目标技术栈是否已有组织约束，例如 Java/Spring、Python/FastAPI、国产化数据库或私有化部署要求。
- 首批接入源优先级：文件上传、数据库同步、消息队列、OA/CRM 哪个先落地。
- 是否需要多租户、文档级权限、段落级权限和权限过滤检索。
- Embedding 已确定采用兼容模型适配：线上通义 text-embedding，本地 BGE；重排模型是否调用外部服务待后续阶段确认。
- 粗排模型采用双塔/DSSM/FM/DNN 训练，还是先使用向量内积或规则分数作为 MVP 方案。
- 向量库是否已有企业标准选型。
