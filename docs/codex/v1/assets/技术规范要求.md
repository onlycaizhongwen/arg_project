## 2. 项目背景与目标 (Why)

### 2.1 现状与痛点
*   **业务现状**：简述客户当前的业务场景（例如：拥有大量非结构化文档，依赖人工查找）。
*   **核心痛点**：
    1.  **信息检索难**：传统关键词搜索无法理解语义，召回率低。
    2.  **数据孤岛**：知识分散在不同系统中，未形成统一资产。
    3.  **响应滞后**：人工处理耗时过长，无法满足实时业务需求。

### 2.2 建设目标
*   **核心指标**：构建一个基于 RAG（检索增强生成）的智能问答系统。
*   **量化价值**：实现知识库检索准确率 > 90%，平均响应时间 < 2s，降低 50% 的人工查询成本。

## 4. 核心业务流程 (How)

### 4.1 数据流：知识入库流程
1.  **采集**：通过 API 或批量上传接入文档（PDF/Word/Markdown）。
2.  **清洗**：去除乱码、冗余格式，提取纯文本。
3.  **切片**：利用语义分割算法，将长文拆分为语义完整的片段。
4.  **向量化**：调用 Embedding 模型将数据转化为高维向量。
5.  **存储**：写入 Milvus 向量库，并建立与原始文件的映射。

### 4.2 业务流：智能问答流程 (RAG)
1.  **接收请求**：用户输入自然语言问题。
2.  **向量检索**：在 Milvus 中查找 Top-K 相关知识片段。
3.  **大模型合成**：将知识片段作为 Context 注入 Prompt，由 LLM 生成答案。
4.  **溯源返回**：返回答案的同时，附带引用来源文档链接。

## 5. 技术选型与核心优势

### 5.1 核心技术栈
| 类别 | 选型方案 | 选型理由 |
| :--- | :--- | :--- |
| **向量数据库** | **Milvus (Standalone)** | 开源、高性能、社区活跃，适合 MVP 阶段快速验证。 |
| **基础模型** | **Ollama (Qwen/Llama3)** | 支持本地私有化部署，数据不出域，保障安全。 |
| **服务框架** | **FastAPI** | 异步高性能，Python 生态丰富，易于集成 AI 组件。 |
| **中间件** | **Redis / PostgreSQL** | 成熟稳定，分别用于缓存加速和元数据管理。 |

### 5.2 核心优势
*   **沙箱化部署 (Docker)**：所有组件容器化，一键启停，不污染宿主环境，便于迁移。
*   **混合检索策略**：结合向量语义检索与关键词检索（BM25），解决单一向量检索的漏召问题。
*   **私有化保障**：全链路本地化，敏感数据不上公有云。

## 6. 部署与实施规划

### 6.1 部署拓扑 (MVP 阶段)
*   **环境要求**：单机 8C16G 及以上。
*   **部署方式**：采用 `Docker Compose` 编排，包含 Milvus、Redis、PostgreSQL 及应用服务。
*   **演进路径**：MVP 验证通过后，可无缝切换至 Kubernetes (K8s) 集群模式以支持高并发。

### 6.2 实施里程碑
| 阶段 | 周期 | 交付物 | 目标 |
| :--- | :--- | :--- | :--- |
| **一期：MVP 验证** | 1-2 周 | Docker 沙箱环境、Demo 演示 | 跑通“文档上传->问答”闭环。 |
| **二期：系统集成** | 2-4 周 | API 接口、管理后台 | 对接客户现有系统，优化召回率。 |
| **三期：上线运维** | 1 周 | 部署手册、培训 | 生产环境上线，建立监控告警。 |

## 7. 质量保障与安全合规

### 7.1 数据安全
*   **数据隔离**：基于租户/用户维度进行数据物理或逻辑隔离。
*   **传输加密**：全链路 HTTPS 传输，防止中间人攻击。
*   **权限控制**：细粒度的 RBAC 权限体系，防止越权访问。

### 7.2 容错与降级
*   **服务降级**：当大模型服务不可用时，自动切换至传统 Elasticsearch 关键词搜索，保证基础可用性。
*   **监控告警**：集成 Prometheus + Grafana，实时监控 GPU 利用率、API 响应时间及错误率。
